# BME-X: What the Paper Says vs What the Code Does

Three-part analysis: (A) the paper's complete story, (B) the code's complete reality, (C) point-by-point contrast.

---

## PART A: WHAT THE PAPER SAYS

### A1. The Problem Statement (p521-522)

The paper identifies four problems with brain MRI:

1. **Motion artifacts** — head movement during scanning creates blurring and ghosting. Young children (2-4 years) only stay still 33-60% of the time.
2. **Low resolution** — hardware limitations, scan time constraints, and participant comfort lead to thick slices and low SNR.
3. **Noise** — thermal noise from coils, stochastic variation, physiological processes. Follows Rician distribution in magnitude images.
4. **Scanner variability** — different manufacturers (Siemens, GE, Philips), field strengths (1.5T vs 3T), and pulse sequences produce different-looking images of the same brain.

The paper criticizes existing methods: prospective methods need extra hardware; retrospective methods need k-space data; deep learning methods (DUNCAN, CycleGAN, Pix2Pix) ignore brain anatomy and produce fake structures; harmonization methods (ComBat) depend on specific batch configurations.

### A2. The Core Theoretical Insight (p534)

The paper's central argument is a **computational complexity reduction**:

Direct mapping from corrupted image to clean image is a many-to-many problem with complexity O(m × n), where m and n are the intensity ranges of the input and output. Every input voxel intensity could map to any output intensity.

BME-X splits this into two simpler steps:
- **Classification**: f₁: I → Lc maps input to c tissue classes. Complexity O(m).
- **Enhancement**: f₂: Lc → I⁰ maps tissue labels to clean image. Complexity O(n).
- **Total**: O(m + n) ≪ O(m × n).

This is the theoretical contribution: the complicated reconstruction task becomes two manageable tasks.

### A3. The Architecture (p534, Fig. 1a)

The paper describes two networks:

**Classification network:**
- DU-Net architecture (ref 77: densely connected U-Net)
- Encoder: three groups, each containing "dense block + Conv + batch normalization (BN) + rectified linear unit (ReLU)"
- A dense block connects encoder and decoder
- Decoder: three groups of "Deconv + BN + ReLU + dense block"
- Final: "a Conv automatically generates class probabilities for each voxel"
- Loss: cross-entropy L₁ = −(1/N) Σᵢ Σⱼ xⱼ(i) ln x̂ⱼ(i) [Eq. 2]

**Connection:**
- "The automated classification output from the classification module is concatenated with the low-quality input images to form a joint input for the enhancement module"
- "To ensure the compatibility of the concatenated data, we apply a Conv + BN + ReLu block to pre-process both the tissue classification and intensity image data"

**Enhancement network:**
- Same DU-Net architecture as classification
- "At the end of the network, a Conv layer is used to generate the high-quality image output"
- Loss: MSE L₂ = (1/N) Σᵢ (y(i) − ŷ(i))² [Eq. 3]

**Joint training:**
- L = L₁ + λL₂, where λ = 10⁻⁷
- "The classification and enhancement networks were jointly trained using the Caffe deep learning framework (Caffe 1.0.0-rc3)"

### A4. Training Details (p534-535)

**Data:**
- 52 fetal participants (21-36 gestational weeks) + 464 participants (0-6 years) from BCP
- All normally developing, artefact-free by visual inspection
- Tissue labels generated by iBEAT V2.0 (intensity inhomogeneity correction, skull stripping, cerebellum removal, tissue segmentation)
- Artefact-free images degraded with simulated motion + blurring to create paired training data

**Degradation simulation:**
- Motion via image-based simulation tool (ref 60): amplitude [0.01, 2.5], frequency [0, 0.0075] × TR, rotation/periodic/continuous/sudden types
- Blurring via artefact simulator (ref 79): Fourier transform → random phase shifts → inverse FFT. σ randomly from [1, 2] (high contrast) or [0.5, 1.5] (low contrast)
- Degradation applied to original brain images before skull stripping

**Configuration:**
- SGD optimizer, LR = 0.005, ×0.1 after each epoch
- Xavier initialization
- Trained for 5 epochs
- 2,000 patches per image, size 40 × 40 × 40, resolution 0.8 × 0.8 × 0.8 mm³
- 95% training, 5% validation
- Separate models for each age group: fetal, 0m, 3m, 6m, 9m, 12m, 18m, 24+ months

**Age-specific models:**
- Brain images at each age group have consistent appearances within group but differ across groups
- For participants 24+ months to 100 years: same model (24+ month)

### A5. What the Paper Claims About Inference

The paper does NOT describe an inference preprocessing pipeline. Fig. 1a shows: Low-quality image → Classification model → Tissue classification map → Enhancement model → High-quality image. No intermediate steps shown.

The paper mentions iBEAT V2.0 only in the context of generating training data ground truth labels. It does not state that iBEAT preprocessing (or equivalent) is required at inference time.

### A6. Key Results Claims

- Outperforms DUNCAN, Pix2Pix, CycleGAN, DU-Net on all metrics (P < 0.001)
- Works across 19 datasets, 3 scanner manufacturers, fetal to elderly
- Preserves tissue volumes (Cohen's d < 0.2 vs STAND images)
- Handles pathological brains (MS lesions, gliomas)
- Generates 7T-like images from 3T
- "Maps all scans, regardless of scanners or sites, to the same common space" (harmonization)

## PART B: WHAT THE CODE ACTUALLY DOES

### B1. Three Separate Codebases

The GitHub repo (https://github.com/DBC-Lab/Brain_MRI_Enhancement) contains THREE unrelated codebases:

1. **`Training_files/`** — Caffe prototxt (5,495 lines). The actual training definition used for the paper's results.
2. **`BME_X/`** — PyTorch inference port. What users download and run. Different architecture than Caffe.
3. **Root-level** (`trainer.py`, `networks/net.py`, `layers.py`) — A completely different model: ViT (Vision Transformer) + MONAI UNet + VoxelMorph deformable registration. NOT BME-X. Never mentioned in the paper.

The Docker container (`yuesun814/bme-x:v1.0.5`, 6.27 GB) contains the production code.

### B2. The Actual Architecture (Container Code)

**Skull stripping model** (`/DUNet3D.py`):
- DU-Net with 32 initial channels, 8 growth rate per dense layer
- Channels: 32→(32+8=40)→(40+8=48)→(48+8=56)→transition...→4 output channels
- BN + ReLU after every conv, Dropout(0.1) at every dense connection
- Output: softmax → 4 tissue probabilities
- Used only for skull stripping, NOT for enhancement
- Model file: `/Model/T1T2w-model.pt` (33 MB)

**Enhancement model** (`/BME_X/models/DUNet3D_seg_recon_softmax.py`):
- DU-Net with **64** initial channels, **16** growth rate — 6.6× more parameters
- Channels: 64→(64+16=80)→(80+16=96)→(96+16=112)→transition...
- Same dense connection pattern: Conv→Dropout(0.1)→Concat→BN→ReLU
- Two identical DU-Nets in one `forward()`: segmentation + reconstruction
- Connection: `softmax(seg_output)` → Conv(4→32)+BN+ReLU, `original_image` → Conv(1→32)+BN+ReLU, concatenate → 64ch → reconstruction DU-Net
- Reconstruction output: Conv(112→1) + BN + ReLU (clips negative values)
- Returns both `(seg_logits, recon_image)`
- Model files: `/BME_X/checkpoints/BMEX-Month{N}-T{1,2}-model.pt` (132 MB each, 16 models)

### B3. The Actual Training (Caffe Prototxt)

**Architecture (from train.prototxt):**
- Segmentation DU-Net: **32** initial channels, **8** growth rate (~2.5M params)
- Reconstruction DU-Net: identical structure (~2.5M params)
- Connection: Conv(4→32)+BN+Scale+ReLU on seg output (which is BN+Scale+ReLU'd logits, NOT softmax)
- Conv(1→32)+BN+Scale+ReLU on original image → Concat → 64ch → recon
- Total: ~5.5M parameters

**Losses:**
- `SoftmaxWithLoss` on segmentation output (weight 1.0) [line 2698]
- `EuclideanLoss` on reconstruction output (weight 0.0000001) [line 5470]
- Caffe EuclideanLoss formula: (1/**2**N) Σ(y−ŷ)², NOT (1/N) as paper states

**Optimizer:**
- SGD, base_lr: 0.005, momentum: 0.9, weight_decay: 0.0005
- lr_policy: "step", gamma: 0.1, stepsize: 222
- max_iter: 5,000,000 (NOT 5 epochs — 22,500 "epochs" by stepsize definition)
- Xavier initialization (Caffe matches paper; PyTorch uses Normal(0, 0.02))

**Data:**
- HDF5 input with 3 blobs: dataT1 (corrupted), dataT1GroundTruth (clean), dataSeg (tissue labels)
- Batch size: 3, iter_size: 3 (effective batch 9)
- Patches from HDF5 (size not verifiable from prototxt — stored in data)

### B4. The Actual Inference Pipeline (Container `/BME-X.py` + `/BME_X/BME_X_enhanced.py`)

The production code executes a **14-step pipeline** for every input image:

**Stage 1: Preprocessing (`/BME-X.py`)**

1. **Intensity rescaling**: Percentile clipping at [0.001, 99.999], then `sitk.IntensityWindowing` to scale intensities to [0, 1000] range.

2. **N4 bias field correction** (first time): `N4BiasFieldCorrectionImageFilter` with `[50, 50, 30, 20]` iterations on the full-head image. Removes RF coil intensity inhomogeneity.

3. **Downsample to 1.6mm isotropic**: For computational efficiency during skull stripping. `sitk.ResampleImageFilter` with linear interpolation.

4. **Histogram matching** (first time): Match to `/Template/template.hdr` (a single generic template). 1024 histogram levels, 7 match points, `ThresholdAtMeanIntensityOn()`.

5. **Skull stripping**: MONAI `sliding_window_inference` with the 32-channel DUNet3D model. ROI 64³, overlap 0.85, input normalized by /10000.0. Produces 4-class tissue probabilities. Brain mask = GM + WM channels merged, binary morphological opening (kernel 3), largest connected component extraction, hole filling.

6. **Upsample mask to original resolution**: Trilinear interpolation back to original image space.

**Stage 2: Enhancement (`/BME_X/BME_X_enhanced.py`)**

7. **Apply brain mask**: `mrimg = mrimg_ori * maskimg` — zero out non-brain voxels.

8. **N4 bias field correction** (second time): Same N4 parameters, applied to the skull-stripped brain.

9. **Resample to model resolution**: 0.8mm isotropic for postnatal, **0.75mm for fetal** (this fetal exception is undisclosed in the paper).

10. **Histogram matching** (second time): Match to age-specific template (`/BME_X/Templates/Template-Month{N}-{T1w|T2w}.nii.gz`). Same parameters: 1024 levels, 7 match points.

11. **Sliding window enhancement**: Patch size **40×40×40**, stride **18**, margin **5** (only center 30³ of each patch is used). 4-class tissue segmentation + 1-channel reconstruction in same forward pass. Overlapping predictions averaged.

12. **Resample back to original resolution**: Both tissue labels and reconstruction resampled to original space.

13. **Reverse histogram matching**: `hist_match(result_nii, filenameT1_forhist)` — the enhanced brain is histogram-matched BACK to the original input's intensity distribution, undoing the standardization from step 10.

14. **Skull compositing**: `skull_img = mrimg_ori * (1 - maskimg)` then `matRecon_withSkull = skull_img + matched_matRecon_array` — enhanced brain pasted back into original skull.

**Stage 3: Quality metrics**

15. **QI calculation**: `QI = TCT_original / TCT_enhanced` where TCT = |μ_wm − μ_gm| / √(σ²_wm + σ²_gm). Written to `{stem}-QI.txt`. This is a RATIO, not a score — higher QI means worse original quality.

### B5. Age-to-Model Mapping (from container code)

```
age >= 21 months  → BMEX-Month24-{T1,T2}-model.pt
15-20 months      → BMEX-Month18
10-14 months      → BMEX-Month12
7-9 months        → BMEX-Month9
5-6 months        → BMEX-Month6
2-4 months        → BMEX-Month3
0-1 months        → BMEX-Month0
< 0 (fetal)       → BMEX-Fetal-T2-model.pt (BROKEN — case mismatch bug)
```

### B6. What the Code Does NOT Do

- **No data augmentation** during training (no flips, rotations, scaling, gamma, brightness)
- **No validation/early stopping** visible in any codebase
- **No dropout during inference** (model set to `eval()` mode)

## PART C: POINT-BY-POINT CONTRAST

### C1. The O(m+n) Complexity Argument

| | Paper | Code |
|---|-------|------|
| **Claim** | Tissue classification simplifies O(m×n) to O(m+n) | The reconstruction loss weight is 10⁻⁷, contributing ~0.001% of gradient. The reconstruction network is almost entirely trained by classification gradient backflow — so the "two separate tasks" are really one task (classification) driving everything |
| **Implication** | Two equally important tasks cooperating | Classification is the sole driver; reconstruction is along for the ride |

### C2. The Architecture

| | Paper | Caffe (training) | PyTorch (inference) |
|---|-------|-----------------|---------------------|
| Initial channels | Not specified | 32 | 64 |
| Dense growth rate | Not specified | 8 | 16 |
| Total params (seg+recon) | Not specified | ~5.5M | ~33M |
| Connection input | "classification output" | ReLU'd logits (raw, unbounded) | softmax probabilities ([0,1], sum=1) |
| Recon output | "a Conv layer" | Conv → BN → Scale → ReLU | Conv → BN → ReLU |
| Dropout | Not mentioned | 0.1 throughout | 0.1 throughout |

**Key difference:** The model users download and run (PyTorch, 33M params) is 6× larger than the model that generated the paper's results (Caffe, 5.5M params). The connection block receives fundamentally different information (probabilities vs logits).

### C3. The Loss Functions

| | Paper | Code |
|---|-------|------|
| CE loss L₁ | −(1/N) Σ xⱼ ln x̂ⱼ | `SoftmaxWithLoss` — standard CE. MATCH. |
| MSE loss L₂ | (1/N) Σ (y−ŷ)² | Caffe `EuclideanLoss` = (1/**2N**) Σ (y−ŷ)². **Factor of 2 difference.** |
| λ = 10⁻⁷ | Yes | `loss_weight: 0.0000001`. MATCH. |
| Joint training | L₁ + λL₂ | Both losses present in Caffe prototxt. MATCH. |

**Practical impact of the 1/2 factor:** The MSE gradient is `(y−ŷ)/N` in Caffe vs `2(y−ŷ)/N` from the paper's formula. Combined with λ=10⁻⁷, the reconstruction gradient is already negligible (~0.001% of total). Halving it again makes it ~0.0005%. Reconstruction quality comes almost entirely from classification gradient backflow.

### C4. Training Configuration

| | Paper | Code | Match? |
|---|-------|------|--------|
| Optimizer | SGD | SGD (Caffe default) | YES |
| LR | 0.005 | base_lr: 0.005 | YES |
| LR decay | ×0.1 per epoch | gamma: 0.1, stepsize: 222 | YES |
| Weight init | Xavier | Caffe: Xavier. PyTorch: Normal(0, 0.02) | Caffe YES, PyTorch NO |
| Epochs | 5 | max_iter: 5,000,000 = 22,500 "epochs" | NO — training continues 4,500× longer than stated |
| Patch size | 40³ | Container: d1=d2=d3=40. GitHub: patchsize=32 | Container YES, GitHub NO |
| Resolution | 0.8mm | 0.8mm postnatal, **0.75mm fetal** | Partial — fetal exception undisclosed |
| Train/val | 95/5 | test_iter=100 in Caffe; no early stopping visible | CANNOT VERIFY |
| Augmentation | Not mentioned | None in any codebase | Consistent (both silent) |

### C5. Inference: The Critical Discrepancy

**What the paper implies (Fig. 1a):**
```
Corrupted image → [Classification model] → Tissue map → [Enhancement model] → Clean image
```
One step. No preprocessing described for inference beyond the model itself.

**What the code actually does:**
```
Raw image
  → Intensity rescale to [0, 1000]
  → N4 bias correction (1st)
  → Downsample to 1.6mm
  → Histogram match to generic template
  → Skull stripping (separate DUNet3D model, MONAI sliding window)
  → Upsample mask to original
  → Apply mask (zero non-brain)
  → N4 bias correction (2nd, on brain only)
  → Resample to 0.8mm
  → Histogram match to age-specific template
  → Enhancement (DUNet3D seg+recon, patch 40³, stride 18, margin 5)
  → Resample back to original
  → REVERSE histogram match (undo standardization)
  → Paste enhanced brain back into original skull
  → Calculate QI
```
Fourteen steps. The model itself is step 11 of 14.

**What this means:** The paper presents a model. The production system is a full preprocessing pipeline with a model in the middle. The preprocessing (N4 twice, histogram matching twice, skull stripping, resampling) does much of the heavy lifting. N4 alone removes bias field artifacts. Histogram matching alone standardizes intensities. The model gets a pre-cleaned, pre-standardized, skull-stripped, resampled input — a much easier task than the raw corrupted image.

### C6. Harmonization

| | Paper | Code |
|---|-------|------|
| **Claim** | "maps all scans, regardless of scanners or sites, to the same common space" | Step 10 histogram-matches to a common template (standardization). Step 13 REVERSE histogram-matches back to original scanner distribution. |
| **Implication** | The model inherently harmonizes | Harmonization exists only in the intermediate representation. The final output is de-harmonized back to the original scanner's intensity space. |
| **Published results** | Supplementary Fig. 3 shows consistent histograms across scanners | These must show the intermediate (step 11 output), not the final delivered image (step 14 output) |

This is the most scientifically significant discrepancy. The paper's strongest practical claim — that BME-X serves as a harmonization tool — is undermined by the production code explicitly undoing the harmonization in the final step.

### C7. The Fetal Case

| | Paper | Code |
|---|-------|------|
| **Claim** | "foetal participants (21-36 gestational weeks)" included in training; separate fetal model | Fetal model exists (`BMEX-Fetal-T2-model.pt`, 132MB) but has a case-mismatch bug (`'Fetal'` vs `'fetal'`) that prevents it from ever being loaded |
| **Template** | Not specified | Fetal-specific template exists (`Template-Fetal-T2w.nii.gz`) but is never loaded (same bug) |
| **Resolution** | 0.8mm stated | Code uses 0.75mm for fetal — undisclosed |
| **Practical impact** | Paper shows fetal results (Fig. 5b) | Container cannot process fetal data without crashing or using wrong model |

### C8. What Makes BME-X Work Despite These Discrepancies

The system works well because of three things the paper undersells:

1. **The preprocessing is the hero.** N4 removes bias fields. Histogram matching standardizes intensities. Skull stripping removes confounders. By the time the model sees the data, most of the "corruption" is already addressed. The model handles the residual motion artifacts and noise on top of a pre-cleaned input.

2. **Tissue classification IS robust to corruption.** The paper's core insight is correct — even severely corrupted brains preserve tissue boundaries. The classification network works well because tissue identity is a spatially coherent, low-frequency property that survives motion artifacts.

3. **Sliding window averaging is powerful.** 40³ patches with stride 18, margin 5, means each voxel is predicted from many overlapping patches. This averaging suppresses noise, boundary artifacts, and model uncertainty. It's computational brute force that reliably produces smooth, consistent outputs.

### C9. What BME-X Cannot Do (Despite Paper Claims)

1. **Process fetal MRI** — the fetal model is broken, fetal data has no skull to strip, and HASTE stacks are 2D not 3D.
2. **Harmonize in the final output** — reverse histogram matching undoes it.
3. **Work without preprocessing** — the pipeline REQUIRES N4, skull stripping, histogram matching, and resampling. Without these, the model receives data unlike anything it was trained on.
4. **Handle unusual pixel types** — SimpleITK cast errors crash the pipeline on some DICOM-converted files (e.g., vector 16-bit signed int).

## CONCLUSION

BME-X is a solid engineering system that produces good results through a well-designed pipeline of classical preprocessing (N4, histogram matching, skull stripping) combined with a tissue-aware deep learning model. The core scientific insight (tissue classification guides enhancement) is valid and well-demonstrated.

However, the paper presents the model as if it operates in isolation, when in reality it's step 11 of a 14-step pipeline. The preprocessing does the majority of the artifact correction (bias field, intensity standardization, skull removal), leaving the model to handle the residual motion artifacts and noise. The harmonization claim is contradicted by the production code which explicitly de-harmonizes the output. The released PyTorch model has 6× more parameters than the Caffe model that generated the published results. And the fetal model — one of the paper's highlighted capabilities — has a bug that prevents it from loading.

The paper is a strong contribution to the field despite these discrepancies. The theoretical framework is elegant, the validation scale is unprecedented, and the downstream task improvements are real. But anyone attempting to reproduce or build upon this work needs to understand that the "model" is really a pipeline, and the pipeline has undisclosed preprocessing that is critical to the results.
